# WKNDR — Amsterdam Source Map & Wire-Up Plan\n\n_Making the weekend feed varied-by-design and self-running._\n\n**Status:** synthesis of six feasibility sweeps · **Date:** 2026-07-01 · **Owner:** lead eng\n\n---\n\n## 0. North star\n\nShift the variety load off the one non-deterministic engine — Claude `web_search` across 10 hardcoded facets (`adapters/websearch.ts`, claude-haiku) — and onto a **breadth of reliable, keyless, machine-readable sources**, each a distinct variety slice, wired as never-throws RA-style adapters. `web_search` is **demoted** from \"carries all the variety\" to \"serendipity/buzz on 2-3 facets.\"\n\nThe centrepiece is **iamsterdam.com's schema.org Event JSON-LD** (The Feed Factory national aggregator): 1,504 live events, category-namespaced URLs, exact dates, real flyer images, keyless, actively maintained (lastmods dated today). It alone carries **5 of the 9 categories deterministically** and touches 2 more.\n\n**Outcome:** week-over-week the feed is rich and beyond concerts+clubs (pop-ups, markets, exhibitions, theatre, food, community) with no human in the loop, inside the ~$1-2/run budget.\n\n### Honesty note on inputs\nOf the six sweeps handed in, **only `culturalDB` returned real, feasibility-tested data.** The other five (`editorial`, `venues`, `platforms`, `niche`, `integration`) came back as empty/test-stub payloads. This map is therefore built on **one verified new source (iamsterdam)** plus the sources already in the codebase (RA live, RSS keyless, Songkick key-gated). Phase 3 re-commissions the stub sweeps to deepen the still-thin slices.\n\n---\n\n## 1. Dedup design — canonical event identity across sources\n\nThe current `dedupe()` (`pipeline.ts`) is **TITLE-keyed** (`titleKey()`: drop year → substring before first separator → strip city tokens → 28 alnum chars; PASS-1 exact-key merge, PASS-2 ≥10-char prefix-collapse). Correct for keyless/editorial sources; **breaks for structured multi-date sources** because iamsterdam recurring items share ONE title across many dates → TITLE-only dedupe collapses valid distinct-date instances.\n\n### The fix (Phase 0, ~10 lines — HARD PREREQUISITE)\n1. **Stable-id short-circuit:** picks with id `web-ra-*` / `web-iams-*` dedupe on their own id (the slug/id encodes the real event instance). Only keyless/canon/rss/llm picks fall through to `titleKey()`.\n2. **Date-aware reconciliation:** after the id-keyed and title-keyed pools exist, merge a title-keyed pick INTO a structured pick only when their weekend-date token matches — so an RA night ALSO cited by web_search still merges (bumps `buzz`), but the same show on two different weekends does not wrongly fuse.\n3. **Keep existing merge rules:** richer record wins (blurb length + `+50` for image), credits union into `source`, `buzz` = distinct source count, `popularity` = max across merged. **ADD:** when a structured pick merges with a keyless dup, the structured pick's `when`, `link`, and flyer `image` always win (structured facts > paraphrase).\n\n**Why it's a prerequisite:** without it, adding iamsterdam makes the feed WORSE (multi-date collapse, buzz miscount). With it, unioning N sources is safe and `balanceByCategory(perCat 6-8)` keeps any one firehose from drowning the others.\n\n---\n\n## 4. Per-source image strategy\n\nThe card is a tall **800×1200 portrait** filled `cover`; the pipeline **drops any live pick without a real relevant photo**. Each adapter ships the best real image it can; the existing `toPortrait()` + vision-verify + themed-photo-bank chain is the safety net.\n\n- **iamsterdam (web-iams-*):** BEST — `image[0]` from JSON-LD `image[]` (Feed Factory `.webp`, hotlinkable 200) → `toPortrait()`. Real per-event flyers, high pass-rate. Keep the existing `isGoodImage` block on the site-level 'Canal Parade' hero.\n- **RA (web-ra-*):** already handled — first `http` `images[].filename` (real flyer, portrait-friendly).\n- **RSS (rss-*):** rarely carries a per-event image; tagged `verify:true`, relies on downstream og:image pass + themed-photo-bank category fallback. That's why RSS is the floor, not the variety lead.\n- **web_search (web-*):** unchanged — per-pick candidate gather + vision subject-match verify + drop-if-reused-across-≥2 (shared banners).\n\n**General rule for any new adapter:** prefer the source's own per-event structured image (JSON-LD `image[]` / detail-page `og:image`) over a listing banner; run every candidate through `toPortrait()`; let vision-verify + themed-bank be the net. Never invent images.\n\n---\n\n## 5. Phased build plan\n\n### Phase 0 — Dedupe prerequisite\nAdd the stable-id short-circuit + date-aware reconciliation to `pipeline.ts dedupe()` (§3). ~10 lines. **Must land before any new source.**\n\n### Phase 1 — iamsterdam adapter (variety centrepiece)\nBuild `app/scripts/adapters/iamsterdam.ts`, RA-style never-throws → `Pick[]`, id `web-iams-<slug>`, source `'I amsterdam'`:\n- Crawl paginated `/calendar/{category}?page=N` (robots-Allowed), **cap ~8-12 detail fetches per category** (~9 cats → ~80-110 fetches/run, pure parse, no LLM → inside budget).\n- Fetch detail HTML → regex `application/ld+json` blocks → `JSON.parse` → keep `@type==Event` where `startDate/endDate` **span** the target weekend (`upcomingWeekend()`).\n- Map: category from URL namespace/`keywords`; area from `location.addressLocality`/street; price from `offers`; blurb from `htmlToText(description)`; image from `image[0]` via `toPortrait()`.\n- Filter OUT `/en/business/calendar/*`. Decisions for Ness\n\n1. **Accept iamsterdam JSON-LD as the deterministic centrepiece?** → Yes. It's the only live, keyless, per-event-structured NL aggregator verified; schema.org Event is the SEO-stable part sites least like to break; wrapped never-throws so a break degrades to today's behaviour. No documented-API alternative exists to wait for.\n2. **Fix dedupe before wiring iamsterdam?** → Yes, Phase 0. ~10 lines and a hard prerequisite — otherwise multi-date events silently collapse.\n3. **Crawl strategy?** → Paginated-listings-first with a ~8-12 per-category cap. robots-Allowed, category-namespaced, bounds cost AND enforces variety at the fetch layer.\n4. **Demote web_search now or later?** → Later (Phase 2). It carries all the variety today; don't remove the net until deterministic sources visibly replace that coverage. Demote = drop to 2-3 facets, not delete.\n5. **data.amsterdam DSO?** → Drop from the build plan; keep one re-check note for after its ~mid-Sept API change.\n6. **Proceed on one verified source or block for more?** → Proceed. iamsterdam (7 cats) + RA (live) + RSS floor is enough to ship varied-by-design + set-and-forget today; re-commission the other five sweeps as a parallel track for eat/drink/shop.\n\n---\n\n_Adapter contract reminder: one file, fetches a source, returns `Pick[]`, **never throws** (returns `[]` on any failure — the rest of the pipeline carries the feed).
